## Task

## What To Check

### 1. Existing Language Infrastructure

- `lang/ru.json` and `lang/en.json` key coverage.
- `data-i18n` usage in HTML.
- language switch behavior in `js/language.js` and `js/main.js`.
- HTML `lang`, `translate`, `notranslate`, and page title behavior.
- fallback behavior when a key is missing.
- whether pages still contain static visible text outside language files.

### 2. Visible UI Text

Find language surfaces in:

- topbar, menu, footer;
- homepage cards and section headings;
- service pages;
- journal public page;
- admin pages;
- NavDesk main and each separated tool page;
- form labels, buttons, helper text, empty states, validation and success/error statuses.

Classify each surface:

- already key-based;
- hardcoded Russian;
- hardcoded English;
- mixed RU/EN;
- generated by JavaScript;
- generated for print/PDF/share text;
- requires owner wording approval.

### 3. Hidden But User-Facing Text

Inventory:

- `placeholder`;
- `aria-label`;
- `title`;
- `alt`;
- meta title/description;
- Open Graph/Twitter text;
- PDF/print document labels;
- browser notification text;
- share text;
- local status messages;
- modal/disclaimer text;
- error text from API/proxy/fallback.

### 4. NavDesk Special Rules

NavDesk has special localization rules:

- day/night switch must remain icon-driven and not become text-heavy;
- disclaimer must stay localized and must preserve the accepted TTL behavior;
- maritime terms must be checked per locale, not translated literally;
- Russian "собака" / 00:00-04:00 and English `dog watch` / 16:00-20:00 must not be blindly treated as the same UI term;
- internal markers may be neutral, but visible labels need locale-specific approval;
- print/PDF documents are product outputs and must be localized too;
- GPS, UTC, LT, course, speed, position, tide, VHF/UKV, and route terms need a controlled glossary.

### 5. Journal And Admin Language Workflow

Confirm the canonical workflow:

- owner writes Russian title, excerpt, body, photo metadata, GPS/media metadata, alt text;
- AI language desk later creates language duplicates through the owner's OpenAI account;
- translations preserve post structure, media order, captions, alt, SEO fields, slugs if approved, statuses, and publication state;
- generated versions are drafts/needs-review until approved;
- no browser-side OpenAI key;
- no manual duplicated RU/EN form model as the long-term solution.

Find all current admin/public surfaces that will need this workflow:

- `admin-posts.html`;
- `js/admin-posts.js`;
- `admin-comments.html`;
- `js/admin-comments.js`;
- `journal.html`;
- `js/journal.js`;
- `data/journal-public.json`;
- live API shape if visible through frontend/proxy docs.
