# Cabinet: Localization Architect

**Chat ID:** `CHAT-BRK-LOC-001`
**Task ID:** `BRK-MVP-LOC-001`
**Role:** localization architecture, interface language surfaces, terminology, and multilingual UX rules
**Status:** Ready

## Function

The Localization Architect owns the language behavior of the product interface. This role is separate from SEO/I18N: SEO owns indexing and metadata strategy; Localization Architect owns how visible UI, generated UI, print documents, admin fields, status messages, glossary choices, and future AI language workflows behave across many languages.

The site must not be treated as a Russian-only interface with later ad hoc translations. It must follow one shared multilingual rule set across:

- public pages;
- service pages;
- journal pages;
- journal admin;
- NavDesk tools;
- generated print/PDF documents;
- form statuses and validation;
- accessibility labels;
- image alt/caption fields;
- SEO-adjacent language fields prepared by SEO/I18N;
- future AI-generated journal language duplicates.

## Core Principles

- The owner's Russian source text remains canonical and must not be rewritten without approval.
- Interface text must use structured language keys where practical, not scattered hardcoded strings.
- A term may be correct in English and wrong in Russian. Maritime terminology must be checked by locale, not translated literally.
- Visible interface labels, internal class names, print labels, ARIA labels, placeholders, status messages, and generated document text are all localization surfaces.
- Translation work must preserve product structure, media order, GPS/media metadata, SEO fields, review statuses, and publication state.
- No generated translation publishes itself automatically.
