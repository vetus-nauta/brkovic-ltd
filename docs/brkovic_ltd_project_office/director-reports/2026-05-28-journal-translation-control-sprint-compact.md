**Owner:** `CHAT-BRK-DIRECTOR-001`  
**Status:** Launched  
**Rule:** no redesign, no production writes, no public URL/SEO changes in this sprint.

## Что закрываем этим спринтом

Собираем оставшиеся микрозадачи по мультиязычной админке журнала в один поток:

1. Убрать эффект “локализация не поддерживается” у AI-кнопок в админке после активации backend route.
2. Привести статус/индикацию AI-потока к стабильному поведению: ошибка/предупреждение/успех.
3. Подтвердить рабочий порядок для `posts` и `journal-collections` с реальным входом в систему.
4. Зафиксировать результат в отчётах и переключить статус задачи `BRK-MVP-BE-005`.

## Набор минизадач

### Backend (через `CHAT-BRK-BACKEND-001`)

`BRK-MVP-BE-005` — **Translation route activation for admin UI**

- Проверить live endpoints:
  - `POST /api/admin/posts/:id/translations/generate`
  - `GET /api/admin/posts/:id/translations`
  - `POST /api/admin/journal-collections/:id/translations/generate`
  - `GET /api/admin/journal-collections/:id/translations`
- Результат при пустой сессии — 401/правильный JSON ошибки.
- Результат при валидной сессии администратора — 200/валидный JSON.
- Зафиксировать статус маршрутов в `backend-admin-translation-route-checklist-2026-05-28.md`.

### Frontend/admin (через `CHAT-BRK-FE-IMPL-001`)

`BRK-MVP-FE-007` и связанный локальный workstream локализации.

- Поддержать стабильный UX состояния AI-кнопок:
  - не показывать “backend не поддерживает” после того, как route стал доступен;
  - считать успехом ответ с валидным `items` по `generate`;
  - отображать warning только при `FAILED`/`skipped` в конкретном языке;
  - не ломать layout (никаких редизайнов карточек).
- Проверить и при необходимости убрать “сиротские” вызовы/обработчики старой кнопки `en из RU` (если она всё ещё всплывает в какой-то ветке).

### QA/UX (через `CHAT-BRK-QA-UX-001`)

- Пробежать smoke админки после логина:
  - открыть один пост и одну коллекцию;
  - `Проверить статусы` оба вида переводов;
  - `Сгенерировать отсутствующие локализации` по нескольким языкам;
  - увидеть зелёный статус только по успешному ответу.
- Зафиксировать наблюдение на 3 вьюпорта:
  - desktop top;
  - tablet;
  - mobile.

### Localization (через `CHAT-BRK-LOC-001`)

`BRK-MVP-LOC-006` и `BRK-MVP-LOC-007`-ветки.

- Сохранить правило: owner пишет RU-сирцевой текст.
- Целевой язык-макет только как roadmap:
 - en, ru, de, it, es, sr, zh
- Не включать UI, где планировщик языков делает “доступными” неподготовленные зоны.

## Порядок шагов

1. Backend подтверждает доступность endpoint-ов: 401 без сессии, 200 с валидной сессией.
2. Frontend привязывает поведение кнопок к фактическим ответам 200/200xx.
3. QA делает 3-хвьюпортный контроль и снимает статус.
4. Director закрывает sprint и открывает следующую фазу (массовая генерация + workflow публикации).
