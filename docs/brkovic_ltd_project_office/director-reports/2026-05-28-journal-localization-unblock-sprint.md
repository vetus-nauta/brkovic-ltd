# Journal Localization Unblock Sprint - 2026-05-28

**Owner:** `CHAT-BRK-DIRECTOR-001`  
**Status:** Planned / Active  
**Main rule:** no redesign, no production changes until API is stable and verified.

## Почему нужен спринт

Сейчас локальный админ-фронт для судового журнала уже умеет делать запросы на многоязычные маршруты, но серверные endpoints всё ещё возвращают `404`. Из-за этого кнопки AI в админке помечают backend как недоступный.

Цель спринта — закрыть цепочку «ручной ввод RU → AI-поддержка EN/другие языки» без изменения редакторского UX и без публикации автоперезаписей.

## Формула спринта

Все задачи делим на 5 мини-пакетов:

1. **Backend unblock** — дать живые админ-эндпоинты.
2. **Стабилизация контрактов** — JSON-форматы и статусы переводов.
3. **Журнал-многостраничности** — покрыть и `posts`, и `journal-collections`.
4. **Пороговые проверки** — автосборка и живой smoke после изменений.
5. **Подписи в отчёте** — обновить статус блоков и принять решение на следующий шаг.

## Микрозадачи

| № | Слой | Задача | Критерий готовности |
| --- | --- | --- | --- |
| 1 | Backend | Реализовать `POST /api/admin/posts/:id/translations/generate` | Возвращает `200/201`, `success`, `data.items[]`, статусы по каждому языку |
| 2 | Backend | Реализовать `GET /api/admin/posts/:id/translations` | Возвращает `data.translations[]` с `{language,status}` |
| 3 | Backend | Реализовать `POST /api/admin/journal-collections/:id/translations/generate` | Та же схема `items[]` и прием body из фронта |
| 4 | Backend | Реализовать `GET /api/admin/journal-collections/:id/translations` | Возвращает статусы переводов для коллекции |
| 5 | Backend | Поддержать payload из UI: `sourceLanguage`, `targetLanguages`, `includeSeo`, `includeMedia` | Запрос без валидационных ошибок на корректных типах |
| 6 | Backend | Привести статусы к принятым значениям (`DRAFT`, `NEEDS_REVIEW`, `PUBLISHED`, `FAILED`, `MISSING`) | Несовпадения статусов не ломают UI | 
| 7 | Frontend | Проверить поведение кнопок AI после открытия маршрутов | После 200-ответа кнопки становятся активны, красный статус API-маршрута исчезает |
| 8 | Frontend | Уточнить тексты статусов (`FAILED`, `skipped`, warning) | Не меняется макет, только корректность сообщений | 
| 9 | QA/UX | Прогнать live smoke на 4 endpoints для одного поста и одной коллекции | Только после авторизованной сессии, HTTP 200 и валидный JSON |
|10 | QA/UX | Обновить статус `BRK-MVP-BE-005` и отчёты | Блокирующий статус в registry должен уйти с `Blocked` после smoke | 

## Технический контракт (единый для всех 4 маршрутов)

- `sourceLanguage`: язык источника (по умолчанию `ru`).
- `targetLanguages`: массив целевых языков.
- `includeSeo`, `includeMedia`: флаги, допустимо игнорировать, но не валидировать строго.
- `POST generate` возвращает:

```json
{
  "success": true,
  "data": {
    "items": [
      { "language": "en", "status": "DRAFT", "skipped": false }
    ]
  }
}
```

- `GET translations` возвращает:

```json
{
  "success": true,
  "data": {
    "translations": [
      { "language": "en", "status": "DRAFT" }
    ]
  }
}
```

## Stop rules

- не пускать автопубликацию переводов;
- не генерировать/публиковать в production без ручной валидации;
- не менять русский текст и не трогать SEO/URL до отдельного спринта;
- не менять визуальный дизайн админки, только состояние состояний/ошибок.

## Definition of Done

- все 4 endpoint-а возвращают ожидаемый JSON и код успеха;
- локальная админка снимает красный статус «backend route недоступен»;
- зелёный статус показывается только после валидного успешного ответа;
- в `task-registry.md` и соответствующем отчёте `BRK-MVP-BE-005` статус сменён с `Blocked` на `For Review` (после подтверждения QA).
