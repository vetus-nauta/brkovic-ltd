## Title, Description, Canonical

Strongest current area:

- Service pages have page-specific title and meta description.
- Service pages have canonical URLs to `https://brkovic.ltd/services/...`.
- Service page meta title/description keys exist in both RU and EN JSON.

Gaps before SEO gate:

1. `index.html` has no canonical.
2. `journal.html` has no canonical and no `data-i18n-title` / `data-i18n-description`, so `js/language.js` can replace the page title with the generic `meta_title` after load.
3. NavDesk pages have `data-i18n-title`, but no canonical and no dynamic `data-i18n-description`; several static descriptions mix English page state with Russian content.
4. No public page has Open Graph or Twitter Card metadata.
5. No public page has JSON-LD structured data.
6. No public page has hreflang alternates.

MVP blocker level:

- Not a hard functional blocker for local MVP upload.
- It is a release SEO risk: public snippets and social previews will be weak, and multilingual indexing is not ready.

## Sitemap And Robots

`sitemap.xml` contains 8 intended public URLs:

- root
- six service pages
- `journal.html`

The public service inventory is covered. NavDesk pages are intentionally absent or undecided; Director should decide whether NavDesk is indexable content or a utility surface kept out of SEO for MVP.

Important sitemap issue:

- Current namespace is `https://www.sitemaps.org/schemas/sitemap/0.9`.
- Standard sitemap namespace is normally `http://www.sitemaps.org/schemas/sitemap/0.9`.
- This should be corrected before production indexing if Director approves a small SEO fix.

`robots.txt`:

- Allows all by default.
- Points to `https://brkovic.ltd/sitemap.xml`.
- Disallows admin pages and `docs/proforma/`.

Robots gap:

- `tools/device-preview.html`, `ops-login.html`, `ops/index.html`, `test-results/`, `game.brkovic.ltd/` and local office docs must not be uploaded as part of main site release. If any operational page is intentionally uploaded, add `noindex` and/or robots exclusions.

## Hreflang And Language Switch

Current language model:

- `js/language.js` supports only `ru` and `en`.
- Initial language is selected from `?lang=`, saved localStorage, or browser language.
- HTML is client-side translated using `lang/ru.json` and `lang/en.json`.
- `document.documentElement.lang` is changed after JS loads.

SEO implication:

- Client-side language switching is useful for users, but it is not enough for final multilingual SEO.
- There are no crawlable language-specific URLs such as `/ru/...` and `/en/...`, and no hreflang alternates.
- Search engines may index the default static HTML and may not treat the RU version as an independent page.

MVP recommendation:

- For release, keep the current switcher as user-facing i18n.
- For real indexing, define URL strategy before adding hreflang: for example `/ru/`, `/en/`, `/ru/services/...`, `/en/services/...`, or static/exported language pages.
- Do not add hreflang until language URLs are actually crawlable and stable.

- `twitter:*`: 0

MVP impact:

- Links shared in messengers/social networks will rely on generic title/description/image detection.
- Service pages and journal would benefit from `og:title`, `og:description`, `og:image`, `og:url`, `og:type`, `twitter:card`.

Release-safe recommendation:

- Add OG/Twitter in a small approved SEO task after Director review.
- Do not rewrite Russian marketing voice while doing this; derive share copy from existing approved titles/descriptions.

## Image Alt And Media SEO

Static images:

- No `<img>` without an `alt` attribute was found in the inspected public pages.
- Empty `alt=""` appears mainly on decorative icons and logo/action icons, which is acceptable when the surrounding link/button has visible text or aria-label.
- Main hero/service images have descriptive alt text.

Journal/admin media:

- Current journal media supports `altRu` and `altEn`.
- Admin media list edits `Alt RU` and `Alt EN`.
- This is sufficient for RU/EN MVP, but it must not be extended by adding `altFr`, `altDe`, etc.
- Future architecture must use `JournalMediaTranslation` or equivalent media/language rows.

SEO risk:

- Public journal entries are JS-rendered, so even good media alt values may not be enough for reliable indexing until public entry pages or static exports exist.

## Headings And Page Intent

Findings:

- Service pages have one H1 each.
- `index.html` has one H1.
- NavDesk pages generally have one H1.
- `journal.html` has 0 H1; visible main title is currently an H2 (`journal_feed_title`).

Recommendation:

- `journal.html` should receive one crawlable H1 in a future approved SEO/markup task.
- Do not redesign the page; this can be a small semantic markup correction.
