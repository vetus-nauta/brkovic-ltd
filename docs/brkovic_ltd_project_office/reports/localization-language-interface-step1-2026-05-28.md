# Отчет: BRK-MVP-LOC-005 — Language Interface Step 1

Дата: 2026-05-28

## 1) Проверочный вывод по модели языков

Исходный контракт языка на уровне интерфейса сейчас соблюдается: `en` помечен как primary, `ru` как дополнительный активный; `de/it/es/sr/zh` — roadmap, но неактивные.

- `en` — `English`
- `ru` — `Русский`
- `de` — `Deutsch`
- `it` — `Italiano`
- `es` — `Español`
- `sr` — `Srpski / crnogorski / hrvatski`
- `zh` — `中文 / Mandarin`

Вердикт по названиям: **в целом приемлемо для roadmap-меню**.

Риски по формулировкам: `sr` и `zh` выглядят как placeholder-агрегаты (см. ниже), но не как ошибки интерфейса.

## 2) Коды: финальные / placeholder

- Финальные (реально активные):
  - `en` (primary, доступный)
  - `ru` (доступный)
- Placeholder / roadmap:
  - `de`, `it`, `es`, `sr`, `zh` (выписаны в `LANGUAGE_OPTIONS`, но `isAvailable: false`)

Дополнение: в коде поддержка загрузки расширяется только для `SUPPORTED_LANGS`, то есть при отсутствии файлов эти языки не доступны.

## 3) Понятность `Primary/Основной` и `Coming/Готовится`

- EN: `Primary`, `Current`, `Coming` — понятны.
- RU: `Основной`, `Текущий`, `Готовится` — понятны и не конфликтуют с задачей roadmap.
- Риск: нужно проверить CSS-состояния отображения меток на кнопках языков (в разметке есть три статических метки-спаны), но это не логический дефект модели, а визуально-интерфейсный риск.

Рекомендуемый текст для стабилизации:
- `Primary`: `Основной (English)` / `Primary (English)`
- `Coming`: `В работе` / `In progress` или оставить `Готовится` как есть.

## 4) Подсказка про системный язык и место смены

Текущая подсказка:
- EN: “Language was selected automatically from your device settings. You can change it in the menu under Language versions.”
- RU: “Язык выбран автоматически по настройкам устройства. Изменить его можно в меню в разделе «Языковые версии».”

Вердикт: **корректная по смыслу**.
- Система указывает автоподбор по устройству (source=`system`).
- Направляет в корректный экран: меню → Language versions.

Ограничение: это всплывающая заметка одноразовая (dismiss с `localStorage`), не влияет на discoverability позже. Нужен отдельный onboarding в меню для пользователей без всплывашки.

## 5) Риски перед URL/SEO решениями для `srb/mne/hr` и Mandarin

### `sr / srb/mne/hr`
- Сейчас это один код `sr` с объединённой маркировкой.
- До решения по региональной стратегии невозможно гарантировать корректный SEO/канонические URL для трех вариантов аудитории (сербская/чорновецкая/хорватская аудитории).
- Риск ложных индексов и смешения контента в выдаче без отдельных локалей/слагов и явной редакторской политике.

### `zh / Mandarin`
- Сейчас один код `zh` без явного разделения `zh-CN` / `zh-TW` / возможного script (`Hans/Hant`).
- До решения по формату сценария выдачи есть риск неоднозначности для пользователей и для будущей индексации.
- `normalizeLang()` принимает только базовый код и не управляет региональными/скриптовыми вариантами через UI.

## 6) Что подготовить до добавления реальных файлов `de/it/es/sr/zh`

1) Языковой контракт
- Зафиксировать для SEO/I18N схему URL и canonical/hreflang.
- Принять финальное решение по `sr` (раздельно или как региональный сводный код) и `zh` (script/регион).

2) Локализационный слой
- Держать `lang/de.json`, `lang/it.json`, `lang/es.json`, `lang/sr.json`, `lang/zh.json` с полной структурой ключей (не partial).
- Проверка полноты ключей между en/ru и новыми локалями (скрипт/чеклист).

3) UI и контентные поверхности
- Вынести/добавить `data-i18n` там, где сейчас есть статичные тексты в:
  - `navdesk.html` (внутренние блоки управления/лейблы/подсказки/шаблоны)
  - `services/yacht-management.html` меню/диспетчер/дисклеймер
- Проверить placeholder/placeholder-строки/alt/title/aria labels, чтобы для новых локалей не было незамещённых английских/русских fallbackов.

4) Процессы и контроль качества
- Согласовать, что публикация переводов идёт через review/publish-гейт и не автоматическая.
- Подготовить QA-пройг для статуса `isAvailable`: кнопки недоступны до присутствия файла и завершённого контента.
