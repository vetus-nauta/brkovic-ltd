## Endpoint Contract Implemented

All routes are under the existing authenticated admin guard:

```text
GET   /api/admin/posts/:id/translations
POST  /api/admin/posts/:id/translations/generate
PATCH /api/admin/posts/:id/translations/:language

GET   /api/admin/journal-collections/:id/translations
POST  /api/admin/journal-collections/:id/translations/generate
PATCH /api/admin/journal-collections/:id/translations/:language
```

Generation writes into existing tables:

```text
JournalTranslation
JournalMediaTranslation
```

Generated rows use:

```text
TranslationStatus.NEEDS_REVIEW
```

Existing `PUBLISHED` translations are not overwritten by generation or patch endpoints.

## Provider And Secret Boundary

OpenAI config is loaded server-side only:

1. `process.env.OPENAI_API_KEY`
2. local fallback `~/.config/brkovic-ltd/openai.env`

The key is not logged, returned, printed, committed, or placed into browser JavaScript.

Default provider mode:

```text
OPENAI_PROVIDER_MODE=stub
```

If `OPENAI_PROVIDER_MODE=live` is set, the first skeleton currently refuses live generation with an intentional error. This keeps the first backend slice safe until prompt/schema validation and owner-approved test content are ready.

## Prisma / Migration Decision

No Prisma migration was added.

Reason:

- `JournalTranslation` already stores title, excerpt, content and SEO draft fields per entity/language.
- `JournalMediaTranslation` already stores media alt/caption per media/language.
- `TranslationStatus.NEEDS_REVIEW` already exists and matches the first review workflow.

Known production gap remains:

- no durable source revision hash field;
- no model/prompt/glossary metadata fields;
- no generated/reviewed/approved timestamps;
- no failed/stale status.

Those need a later migration before production AI editorial use.
