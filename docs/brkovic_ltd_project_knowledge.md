# BRKOVIC.LTD project knowledge

Живая база правил проекта. Ее нужно читать вместе с полным handoff перед продолжением работы над страницами.

## Мобильная компоновка страниц услуг

Решение принято после переработки страницы `services/yacht-acceptance-delivery.html`.

Главный принцип: на мобильных устройствах не убирать смысл и не упрощать услугу до рекламного баннера. Нужно сохранить содержание, но дать пользователю быстрый обзор и убрать длинную "портянку" за счет группировки и раскрываемых блоков.

Применять ко всем новым и обрабатываемым страницам услуг:

- Десктоп и планшет сохраняют полноценную структуру, если она хорошо читается.
- Мобильная версия получает короткий блок "Коротко" сразу после hero.
- Длинные информационные карточки на мобильном превращаются в вертикальные раскрывающиеся секции.
- Секции должны иметь человеческие заголовки, которые сами объясняют пользу: не технические ярлыки и не канцелярит.
- Важные интерактивные инструменты на мобильном показывают сначала основные поля и результат.
- Ручные, уточняющие или экспертные параметры прячутся под раскрытие вроде "Уточнить параметры".
- Не плодить отдельные подстраницы ради короткого мобильного экрана: пользователь теряет контекст.
- Не добавлять вторую нижнюю навигацию там, где уже есть `mobile-dock`.
- Не использовать горизонтальные табы для важного контента на мобильном: вертикальные раскрывающиеся блоки читаются спокойнее.
- Первый экран должен дать ответ "что это и зачем", а не заставлять листать до сути.

Текущий эталон техники:

```text
services/yacht-acceptance-delivery.html
css/main.css
css/responsive.css
js/delivery-calculator.js
```

Что сделано на эталонной странице:

- Добавлен компактный мобильный блок "Коротко".
- Блоки "Что входит", "Кому подходит", "География", "Результат" свернуты на мобильном.
- Условия "Экипаж", "Отдельные расходы", "Маршрут" свернуты на мобильном.
- Калькулятор на мобильном показывает основные поля: тип судна, длина, старт, финиш, расчет.
- Поля "Дистанция", "Цена топлива", "Расход топлива" вынесены в "Уточнить параметры".
- Стартовое значение дистанции в калькуляторе — `0`, чтобы не показывать фиктивную цену до ввода маршрута.

При переносе техники на другие страницы сначала смотреть смысл страницы, а не механически копировать блоки. Страница должна оставаться живой, деловой и понятной именно для своей услуги.

## Единое поведение страниц услуг

Следующий общий проход по сайту должен привести остальные страницы услуг к единому поведению, освоенному на `services/yacht-acceptance-delivery.html` и `services/yacht-registration.html`.

- Хедер на страницах услуг держать по эталону главной: логотип слева, тап по логотипу ведет на главную.
- Кнопку "Все услуги" в hero не возвращать: она выбивает пользователя из воронки. Достаточно CTA на заявку и общей навигации в футере / mobile dock.
- Hero каждой услуги строить вокруг своего персонажа и сцены по смыслу услуги. Это не фотографический двойник владельца, а художественный персонаж в общей традиции сайта.
- Для новых hero внимательно проверять морскую логику: такелаж, положение шкипера, трапы, кранцы, обувь, перспективу причала, направление лодки и реальные детали борта.
- На мобильной версии не делать длинную "портянку": быстрый обзор, затем раскрываемые смысловые блоки.
- На десктопе и планшете сохранять плотную, спокойную деловую компоновку, если она читается хорошо.
- Футер оставлять картой сайта с человеческими названиями ссылок. На страницах с админ-ссылками ничего не сносить без проверки.

## Главная страница и карточки услуг

Главную страницу нужно переосмыслить после наполнения страниц услуг: она должна вести человека в реальные сервисы, а не жить отдельно.

- Погода на главной сейчас выглядит лишней: скорее всего, ее убрать или вынести из первого смыслового слоя.
- Карточки услуг проверить на информативность и мусорность: меньше общих слов, больше понятного повода перейти в услугу.
- Основные карточки услуг выровнять по сетке. Сейчас `Yacht Management` выбивается шириной и ломает общий ритм.
- Полностью синяя карточка management слишком тяжелая и мешает восприятию остальных услуг. Ее нужно ослабить.
- За визуальный принцип взять карточку IYT: неброский тематический оттенок, маленькие логотипы/метки, аккуратная тень, без пестроты.
- Для остальных карточек подумать о тихих тематических отголосках: пальмы или вода для отдыха, документы для регистрации, чек-лист/корпус для приемки, маршрутная линия для доставки, навигационный намек для шкипера.
- Тематические элементы должны быть блеклыми и вторичными, например в углу карточки. Не превращать главную в набор разных плакатов.

## Backend и FTP судового журнала

Backend судового журнала не находится внутри frontend-репозитория `brkovic-ltd`, но он найден на боевом сервере:

```text
/journal-backend
```

На этом Linux-ПК FTP-доступ настроен локально вне Git через `~/.netrc`. Секреты не коммитить и не переносить в документы проекта.

Для следующих чатов:

- читать `docs/brkovic_ltd_backend_ftp_access_notes_2026-05-25.md`;
- читать `docs/brkovic_ltd_journal_audit_2026-05-25.md`;
- локальный read-only snapshot backend-а лежит в `/home/alexey/.local/share/brkovic-ltd/ftp-snapshot/journal-backend`;
- локальная working copy backend-а лежит в `/home/alexey/.local/share/brkovic-ltd/work/journal-backend`;
- backend: NestJS + Prisma + PostgreSQL;
- production DB/backend уже получили multipage / collections / translations schema и collection API 2026-05-25;
- public `/api/public/journal/collections` работает и пока возвращает пустой массив;
- public like / unlike / like-status для обычных записей подключены;
- PostgreSQL напрямую с локального ПК не открыт, база доступна backend-у на сервере через localhost;
- перед следующими backend/schema-изменениями снова нужен свежий backup `/journal-backend` и релевантных таблиц базы.

## OpenAI API и многоязычный журнал

У владельца есть активный OpenAI API key. Секрет нельзя вставлять в чат, Git, HTML, JS, `lang/*.json`, Google Drive или офисные отчеты.

Правило проекта:

- OpenAI используется только server-side;
- браузерная админка не видит ключ;
- AI language desk создает черновики переводов и SEO, но не публикует автоматически;
- русский пост владельца остается canonical source;
- итоговые языковые версии проходят review/approve/publish.

Локально ключ хранить вне репозитория:

```text
~/.config/brkovic-ltd/openai.env
```

Статус на `Vetus-Home`: локальный `openai.env` создан, права `600`, значение ключа не выводилось и не записывалось в проект.

Для production использовать environment variable или приватный путь вне public root, например:

```text
/home/brkovic/private/openai.env
```

См.:

```text
docs/brkovic_ltd_project_office/reports/openai-api-key-server-boundary-2026-05-27.md
docs/brkovic_ltd_project_office/cabinets/backend-admin/task-0002-openai-language-desk-server-boundary.md
api/openai.config.example.php
```

## Целевая языковая модель проекта

Директорское уточнение владельца от 2026-05-28 важнее старых черновых упоминаний только `ru/en` и старого future-list с `fr`.

Целевая модель публичного сайта:

- основной язык сайта: `en` / English;
- дополнительные языковые версии: `ru`, `de`, `it`, `es`, общий региональный `srb/mne/hr`, Mandarin;
- для регионального `srb/mne/hr` временный технический код в frontend-roadmap: `sr`, пока SEO/Localization/Backend не утвердят финальные URL, локаль и разделение/объединение вариантов;
- французский `fr`, найденный в старом handoff как future planned, не считать актуальным без отдельного возврата владельцем;
- меню не должно возвращаться к короткому `RU/EN`: оно должно отражать многоязычную модель, даже если сегодня реально подключены только `en` и `ru`.

Важное разделение:

- English - основной публичный язык сайта и будущая базовая версия для SEO-выхода;
- русский авторский текст судового журнала остается рабочим исходником владельца в админке;
- AI language desk создает черновики переводов и SEO для целевых языков, но не публикует их автоматически.

Текущий frontend имеет только `lang/en.json` и `lang/ru.json`. Остальные языки в меню можно показывать как "готовится", но нельзя пытаться грузить отсутствующие JSON-файлы или добавлять `hreflang` до появления реальных crawlable URL.

Текущий директорский приказ по MVP:

```text
docs/brkovic_ltd_project_office/director-reports/2026-05-27-mvp-stabilization-order.md
```

Фаза сейчас - стабилизация MVP, не редизайн. Интерфейс не переделывать без согласования владельца/директора. UX только фиксирует блокеры, риски и предложения, помечая их как требующие согласования. Доделывание перед релизом начинается только в функциях `NavDesk`, если не будет отдельной новой задачи.

Финальный локальный MVP gate-review от 2026-05-27:

```text
docs/brkovic_ltd_project_office/director-reports/2026-05-27-final-mvp-gate-review.md
```

Вывод: локальный MVP-аудит завершен, явный MVP-блокер не найден, но production upload не запущен автоматически. Следующий риск - дисциплина выкладки: точный пакет, backup production, сохранение server-only config и production data. Малые SEO-правки перед upload возможны только отдельным решением владельца/директора.

## Обязательный маркер: многоязычный журнал через AI

Многоязычность судового журнала не должна потеряться в MVP-аудите. Это отдельная архитектурная задача офиса:

```text
docs/brkovic_ltd_project_office/director-reports/2026-05-27-ai-multilingual-admin-marker.md
docs/brkovic_ltd_project_office/cabinets/seo-i18n/task-0002-ai-multilingual-admin-architecture.md
```

Канонический сценарий: владелец пишет запись в админке на русском, добавляет фото, геометки, подписи и alt. Дальше отдельный AI-раздел админки через OpenAI-аккаунт владельца создает языковые дубли с сохранением структуры, медиа, SEO и статусов проверки. Генерация переводов не публикует их автоматически и не заменяет авторский русский текст.

Технический принцип: не плодить поля под каждый язык. Для журнала нужна масштабируемая модель переводов через `JournalTranslation` / `JournalMediaTranslation` или совместимую таблицу строк по entity/language. OpenAI-ключи хранятся только server-side, не в браузере и не в Git.

## Должность Localization Architect

Добавлена отдельная должность:

```text
docs/brkovic_ltd_project_office/cabinets/localization-architect/
docs/brkovic_ltd_project_office/cabinets/localization-architect/task-0001-language-surface-inventory.md
```

Причина: многоязычность проекта касается не только SEO и не только русского/английского JSON. Локализатор должен отдельно проверить видимые строки, скрытые подписи, `placeholder`, `aria-label`, `alt`, meta, печать/PDF/share/notifications, JS-generated statuses, NavDesk terminology, журнал/admin workflow и будущие AI-дубли по многим языкам. Правки локализации начинать только после отчета-инвентаризации и решения владельца/директора.

## Должность SEO Integration Director

Для боевой SEO-интеграции создан отдельный директорский кабинет:

```text
docs/brkovic_ltd_project_office/cabinets/seo-integration-director/
docs/brkovic_ltd_project_office/cabinets/seo-integration-director/new-chat-prompt.md
```

`seo-i18n` остается аудитом, языками, metadata и архитектурным маркером. `SEO Integration Director` - должность выше по уровню: стратегия и тактика выхода в живой поиск, page-agent command, index/noindex политика, schema, OG/Twitter, sitemap/robots, внутренние связи, журнал, NavDesk и многоязычные SEO-гейты. Страницы должны разбираться отдельно, потому что запросы, интент и полезность у главной, услуг, журнала и NavDesk разные.
